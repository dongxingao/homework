# Mini-12306 数据库设计

## 1. 设计目标

数据库设计的目标是为 Mini-12306 系统提供稳定、清晰、可维护的数据存储结构，支撑以下核心业务：

- 用户注册与登录
- 车次查询
- 在线购票
- 查看订单
- 在线退票
- 在线改签
- 管理员维护车次信息

本项目为课程实践项目，第一版数据库设计遵循“结构尽量简单、能够支撑核心流程、便于后续扩展”的原则。

## 2. 设计原则

本系统数据库设计遵循以下原则：

- 满足当前核心业务需求，优先保证购票流程完整。
- 表结构尽量简单，便于前后端开发和联调。
- 关键字段设置必要的唯一约束和非空约束。
- 使用主键和外键表达实体之间的关联关系。
- 对常用查询字段建立索引，提高查询效率。
- 为后续扩展支付、乘车人管理、日志记录等功能保留空间。

## 3. 数据库概念设计

### 3.1 核心实体

根据需求分析和用例分析，系统主要包含以下三个核心实体：

- `用户（users）`
- `车次（trains）`
- `订单（orders）`

### 3.2 实体关系分析

实体之间的关系如下：

- 一个用户可以拥有多个订单，因此 `users` 与 `orders` 是一对多关系。
- 一个车次可以对应多个订单，因此 `trains` 与 `orders` 是一对多关系。
- 每个订单只能属于一个用户，也只能对应一个车次。

### 3.3 E-R 关系文字说明

可以将系统 E-R 图抽象为：

- 用户 `1 ---- n` 订单
- 车次 `1 ---- n` 订单

也就是说，订单表是连接用户和车次的核心业务表。

## 4. 逻辑结构设计

### 4.1 用户表 `users`

用户表用于存储系统用户的基本信息、登录信息和角色信息。

| 字段名 | 数据类型 | 是否主键 | 是否可空 | 约束说明 | 说明 |
|---|---|---|---|---|---|
| id | BIGINT | 是 | 否 | 自增主键 | 用户 ID |
| username | VARCHAR(50) | 否 | 否 | 唯一 | 用户名 |
| password | VARCHAR(255) | 否 | 否 |  | 登录密码 |
| real_name | VARCHAR(50) | 否 | 否 |  | 真实姓名 |
| id_card | VARCHAR(18) | 否 | 否 | 唯一 | 身份证号 |
| phone | VARCHAR(20) | 否 | 否 | 唯一 | 手机号 |
| role | VARCHAR(20) | 否 | 否 | 默认 `PASSENGER` | 用户角色 |
| created_at | DATETIME | 否 | 否 | 默认当前时间 | 创建时间 |
| updated_at | DATETIME | 否 | 否 | 默认当前时间 | 更新时间 |

#### 字段说明

- `username` 用于登录，必须唯一。
- `password` 存储加密后的密码，实际实现中不应明文保存。
- `role` 用于区分普通旅客和管理员，建议值包括：
  - `PASSENGER`
  - `ADMIN`

### 4.2 车次表 `trains`

车次表用于存储车次基础信息、时间信息、票价和余票信息。

| 字段名 | 数据类型 | 是否主键 | 是否可空 | 约束说明 | 说明 |
|---|---|---|---|---|---|
| id | BIGINT | 是 | 否 | 自增主键 | 车次 ID |
| train_no | VARCHAR(20) | 否 | 否 | 唯一 | 车次编号 |
| from_station | VARCHAR(50) | 否 | 否 |  | 出发站 |
| to_station | VARCHAR(50) | 否 | 否 |  | 到达站 |
| departure_time | DATETIME | 否 | 否 |  | 发车时间 |
| arrival_time | DATETIME | 否 | 否 |  | 到达时间 |
| price | DECIMAL(10,2) | 否 | 否 |  | 票价 |
| total_seats | INT | 否 | 否 |  | 总票数 |
| remaining_seats | INT | 否 | 否 |  | 剩余票数 |
| status | VARCHAR(20) | 否 | 否 | 默认 `AVAILABLE` | 车次状态 |
| created_at | DATETIME | 否 | 否 | 默认当前时间 | 创建时间 |
| updated_at | DATETIME | 否 | 否 | 默认当前时间 | 更新时间 |

#### 字段说明

- `train_no` 是业务上的车次编号，例如 `G123`、`K456`。
- `price` 采用 `DECIMAL(10,2)` 存储，避免浮点误差。
- `remaining_seats` 不能小于 0，也不能大于 `total_seats`。
- `status` 表示车次是否可售，建议值包括：
  - `AVAILABLE`：可售
  - `STOPPED`：停运
  - `CLOSED`：停止售票

### 4.3 订单表 `orders`

订单表用于记录用户购票、退票、改签等业务数据。

| 字段名 | 数据类型 | 是否主键 | 是否可空 | 约束说明 | 说明 |
|---|---|---|---|---|---|
| id | BIGINT | 是 | 否 | 自增主键 | 订单 ID |
| order_no | VARCHAR(50) | 否 | 否 | 唯一 | 订单编号 |
| user_id | BIGINT | 否 | 否 | 外键 | 用户 ID |
| train_id | BIGINT | 否 | 否 | 外键 | 车次 ID |
| passenger_name | VARCHAR(50) | 否 | 否 |  | 乘车人姓名 |
| passenger_id_card | VARCHAR(18) | 否 | 否 |  | 乘车人身份证号 |
| price | DECIMAL(10,2) | 否 | 否 |  | 订单票价 |
| status | VARCHAR(20) | 否 | 否 | 默认 `PAID` | 订单状态 |
| original_order_id | BIGINT | 否 | 是 | 可空外键 | 原订单 ID，用于改签追踪 |
| created_at | DATETIME | 否 | 否 | 默认当前时间 | 下单时间 |
| paid_at | DATETIME | 否 | 是 |  | 支付时间 |
| updated_at | DATETIME | 否 | 否 | 默认当前时间 | 更新时间 |

#### 字段说明

- `order_no` 用于展示给用户查看，必须唯一。
- `user_id` 关联下单用户。
- `train_id` 关联具体车次。
- `passenger_name` 和 `passenger_id_card` 表示乘车人信息。
- `status` 表示订单当前业务状态。
- `original_order_id` 用于记录改签来源，如果该订单是由改签生成，则指向原订单。

#### 订单状态建议

建议订单状态统一使用以下取值：

- `UNPAID`：未支付
- `PAID`：已支付
- `REFUNDED`：已退票
- `CHANGED`：已改签
- `CANCELLED`：已取消

对于第一版简化项目，如果不实现真实支付流程，也可以默认购票成功后直接将订单状态置为 `PAID`。

## 5. 表关系设计

### 5.1 用户表与订单表关系

- 一个用户可以拥有多个订单。
- `orders.user_id` 外键关联 `users.id`。

### 5.2 车次表与订单表关系

- 一个车次可以对应多个订单。
- `orders.train_id` 外键关联 `trains.id`。

### 5.3 改签关系设计

为了支持改签操作，订单表中增加 `original_order_id` 字段。

其作用是：

- 记录当前订单是否由其他订单改签而来。
- 便于后续追踪改签链路。

如果项目实现选择“改签只更新原订单，不生成新订单”，则该字段可以暂时保留但不强制使用。

## 6. 数据完整性约束

系统应满足以下数据完整性要求：

- 用户名不能为空且必须唯一。
- 身份证号不能为空且必须唯一。
- 手机号不能为空且必须唯一。
- 车次编号不能为空且必须唯一。
- 车次票价不能小于 0。
- 总票数不能小于 0。
- 剩余票数不能小于 0，且不能大于总票数。
- 订单必须关联有效的用户和车次。
- 同一订单编号不能重复。

## 7. 索引设计

为了提高系统查询效率，建议建立如下索引：

### 7.1 用户表索引

- `username` 唯一索引
- `id_card` 唯一索引
- `phone` 唯一索引

### 7.2 车次表索引

- `train_no` 唯一索引
- `(from_station, to_station, departure_time)` 组合索引
- `status` 普通索引

### 7.3 订单表索引

- `order_no` 唯一索引
- `user_id` 普通索引
- `train_id` 普通索引
- `status` 普通索引

## 8. 物理设计建议

### 8.1 数据库选型

课程项目建议使用关系型数据库，如：

- MySQL
- MariaDB

如果项目规模较小，也可以在开发阶段使用：

- SQLite

但如果需要体现较完整的工程化设计，推荐优先采用 `MySQL`。

### 8.2 字符集建议

建议数据库统一使用：

- 字符集：`utf8mb4`
- 排序规则：`utf8mb4_general_ci`

这样可以兼容中文站点名称、用户姓名等数据。

### 8.3 主键设计建议

建议所有核心表采用：

- `BIGINT` 自增主键

优点是实现简单、便于调试、适合课程项目。

## 9. 建表 SQL 设计

以下给出基于 MySQL 的建表 SQL 示例。

### 9.1 用户表建表语句

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '用户ID',
    username VARCHAR(50) NOT NULL UNIQUE COMMENT '用户名',
    password VARCHAR(255) NOT NULL COMMENT '密码',
    real_name VARCHAR(50) NOT NULL COMMENT '真实姓名',
    id_card VARCHAR(18) NOT NULL UNIQUE COMMENT '身份证号',
    phone VARCHAR(20) NOT NULL UNIQUE COMMENT '手机号',
    role VARCHAR(20) NOT NULL DEFAULT 'PASSENGER' COMMENT '用户角色',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

### 9.2 车次表建表语句

```sql
CREATE TABLE trains (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '车次ID',
    train_no VARCHAR(20) NOT NULL UNIQUE COMMENT '车次编号',
    from_station VARCHAR(50) NOT NULL COMMENT '出发站',
    to_station VARCHAR(50) NOT NULL COMMENT '到达站',
    departure_time DATETIME NOT NULL COMMENT '发车时间',
    arrival_time DATETIME NOT NULL COMMENT '到达时间',
    price DECIMAL(10,2) NOT NULL COMMENT '票价',
    total_seats INT NOT NULL COMMENT '总票数',
    remaining_seats INT NOT NULL COMMENT '剩余票数',
    status VARCHAR(20) NOT NULL DEFAULT 'AVAILABLE' COMMENT '车次状态',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='车次表';
```

### 9.3 订单表建表语句

```sql
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '订单ID',
    order_no VARCHAR(50) NOT NULL UNIQUE COMMENT '订单编号',
    user_id BIGINT NOT NULL COMMENT '用户ID',
    train_id BIGINT NOT NULL COMMENT '车次ID',
    passenger_name VARCHAR(50) NOT NULL COMMENT '乘车人姓名',
    passenger_id_card VARCHAR(18) NOT NULL COMMENT '乘车人身份证号',
    price DECIMAL(10,2) NOT NULL COMMENT '订单票价',
    status VARCHAR(20) NOT NULL DEFAULT 'PAID' COMMENT '订单状态',
    original_order_id BIGINT NULL COMMENT '原订单ID',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '下单时间',
    paid_at DATETIME NULL COMMENT '支付时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    CONSTRAINT fk_orders_user FOREIGN KEY (user_id) REFERENCES users(id),
    CONSTRAINT fk_orders_train FOREIGN KEY (train_id) REFERENCES trains(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='订单表';
```

### 9.4 索引补充语句

```sql
CREATE INDEX idx_trains_route_time
ON trains(from_station, to_station, departure_time);

CREATE INDEX idx_trains_status
ON trains(status);

CREATE INDEX idx_orders_user_id
ON orders(user_id);

CREATE INDEX idx_orders_train_id
ON orders(train_id);

CREATE INDEX idx_orders_status
ON orders(status);
```

## 10. 数据库设计与业务功能对应关系

数据库表与业务功能的对应关系如下：

- `users`：支撑注册、登录、角色控制
- `trains`：支撑车次查询、管理员维护、余票管理
- `orders`：支撑购票、查看订单、退票、改签

其中：

- 购票操作主要涉及 `orders` 新增和 `trains.remaining_seats` 扣减
- 退票操作主要涉及 `orders.status` 更新和 `trains.remaining_seats` 回补
- 改签操作主要涉及原订单状态变更、新订单生成或订单更新，以及两个车次的余票同步调整

## 11. 后续扩展建议

如果后续需要扩展系统，可考虑增加以下表：

- `stations`：车站表，统一维护站点名称
- `payments`：支付记录表，记录支付流水
- `passengers`：常用乘车人表，支持一个用户管理多个乘车人
- `operation_logs`：操作日志表，记录管理员关键操作
- `notifications`：消息通知表，用于短信或站内消息扩展

当前第一版暂不引入这些表，以控制系统复杂度。

## 12. 小结

本数据库设计以 `users`、`trains`、`orders` 三张核心表为中心，能够支撑 Mini-12306 第一版的主要业务流程。该设计结构清晰、实现成本较低，适合作为课程项目的初版数据库方案。在后续开发中，可以在不破坏当前主结构的前提下，逐步扩展支付、乘车人管理和消息通知等功能。
