# Mini-12306 需求分析

## 1. 引言

### 1.1 项目背景

传统火车票购买、退票和改签主要依赖线下售票窗口，旅客需要到车站排队办理，效率较低，且无法方便、及时地获取车次及余票信息。为提高购票效率、改善用户体验，需要开发一个基于 Web 的简化版铁路票务服务系统 `Mini-12306`。

该系统面向旅客和管理员，提供在线查询车次、购票、退票、改签以及车次管理等功能，实现基础票务业务的数字化处理。

### 1.2 项目目标

Mini-12306 的目标是构建一个最小可运行的在线火车票服务系统，使普通用户可以完成基础购票流程，使管理员可以维护车次和余票信息。

### 1.3 项目范围

本项目第一版主要实现以下内容：

- 用户注册
- 用户登录
- 车次查询
- 在线购票
- 订单查看
- 在线退票
- 在线改签
- 管理员管理车次

本项目第一版暂不实现以下内容：

- 真实身份证联网认证
- 真实短信验证码
- 真实银行支付接口
- 抢票候补功能
- 在线选座功能
- APP 客户端

## 2. 业务概述

### 2.1 目标用户

系统面向以下三类用户：

- 游客：未登录用户，只能浏览基础页面、注册和登录。
- 旅客：已注册并登录的普通用户，可以查询车次、购票、查看订单、退票和改签。
- 管理员：负责维护车次、票价、余票和状态等信息。

### 2.2 业务目标

系统需要解决以下核心问题：

- 让旅客不必到线下窗口排队即可完成票务操作。
- 让旅客能够在线查看车次、发车时间、到达时间、票价和余票。
- 让管理员可以统一维护车次数据，保证系统中的票务信息可管理、可更新。

### 2.3 业务流程概述

旅客的基本业务流程如下：

1. 用户注册并登录系统。
2. 用户输入出发地、目的地和出发日期查询车次。
3. 系统返回符合条件的车次列表及余票信息。
4. 用户选择目标车次并提交购票请求。
5. 系统校验用户状态、车次状态和余票数量。
6. 校验通过后生成订单并完成购票。
7. 用户可在“我的订单”中查看订单，并执行退票或改签操作。

管理员的基本业务流程如下：

1. 管理员登录后台。
2. 管理员新增、修改、删除车次信息。
3. 系统更新车次、票价、时间和余票数据。

## 3. 功能需求分析

### 3.1 用户注册

#### 功能说明

游客填写个人基本信息完成注册，成为系统合法用户。

#### 输入信息

- 用户名
- 密码
- 真实姓名
- 身份证号
- 手机号

#### 处理规则

- 用户名必须唯一。
- 身份证号格式应合法，且同一身份证号不能重复注册。
- 手机号格式应合法。
- 密码不能为空，长度应满足最小安全要求。

#### 输出结果

- 注册成功，提示用户前往登录。
- 注册失败时，返回明确失败原因，如“用户名已存在”。

### 3.2 用户登录

#### 功能说明

已注册用户通过账号和密码登录系统，进入对应角色页面。

#### 输入信息

- 用户名
- 密码

#### 处理规则

- 校验用户名是否存在。
- 校验密码是否正确。
- 根据用户角色分配访问权限。

#### 输出结果

- 登录成功，进入用户首页或管理页面。
- 登录失败，提示用户名或密码错误。

### 3.3 查询车次

#### 功能说明

用户根据出发地、目的地、出发日期等条件查询可用车次。

#### 输入信息

- 出发地
- 目的地
- 出发日期

#### 处理规则

- 系统根据查询条件筛选符合条件的车次。
- 展示车次编号、发车时间、到达时间、票价、余票、状态等信息。
- 若无匹配结果，应返回空列表并提示。

#### 输出结果

- 返回符合条件的车次列表。

### 3.4 在线购票

#### 功能说明

旅客选择车次后提交购票请求，系统完成订单生成和票额扣减。

#### 输入信息

- 车次编号或车次 ID
- 乘车人姓名
- 乘车人身份证号

#### 处理规则

- 用户必须处于登录状态。
- 所选车次必须存在且状态正常。
- 车次余票必须大于 0。
- 购票成功后生成唯一订单编号。
- 购票成功后扣减对应车次的余票数量。

#### 输出结果

- 购票成功，返回订单信息。
- 购票失败，提示失败原因，如“余票不足”。

### 3.5 查看订单

#### 功能说明

旅客查看自己的购票记录和订单状态。

#### 展示内容

- 订单编号
- 车次信息
- 乘车人信息
- 票价
- 订单状态
- 下单时间

#### 输出结果

- 返回当前用户的订单列表。

### 3.6 在线退票

#### 功能说明

旅客对已购票订单发起退票申请，系统完成订单状态更新和余票回补。

#### 输入信息

- 订单 ID 或订单编号

#### 处理规则

- 只能退当前登录用户自己的订单。
- 已退票订单不能重复退票。
- 订单状态符合退票条件时，允许退票。
- 退票成功后，订单状态变为 `REFUNDED`。
- 系统需要将对应车次余票数量加 1。

#### 输出结果

- 退票成功，返回更新后的订单状态。
- 退票失败，提示失败原因。

### 3.7 在线改签

#### 功能说明

旅客可将已购订单改签到其他满足条件的车次。

#### 输入信息

- 原订单 ID
- 新车次 ID

#### 处理规则

- 原订单必须属于当前用户。
- 原订单必须处于可改签状态。
- 目标车次必须存在且余票充足。
- 改签成功后，原订单状态变为 `CHANGED` 或更新为新的有效订单记录。
- 系统需要同步处理原车次与新车次的余票数量。

#### 输出结果

- 改签成功，返回新订单或更新后的订单信息。
- 改签失败，提示失败原因，如“目标车次余票不足”。

### 3.8 管理员管理车次

#### 功能说明

管理员维护车次基础信息，支撑前台查询和购票功能。

#### 子功能

- 新增车次
- 修改车次
- 删除车次
- 调整余票
- 修改车次状态

#### 车次信息字段

- 车次编号
- 出发站
- 到达站
- 发车时间
- 到达时间
- 票价
- 总票数
- 剩余票数
- 车次状态

#### 处理规则

- 仅管理员可访问该功能。
- 被订单引用的车次删除时应进行限制或逻辑删除。
- 剩余票数不能大于总票数，也不能小于 0。

#### 输出结果

- 车次维护成功时返回成功提示。
- 数据不合法时返回明确错误信息。

## 4. 业务规则需求

系统需满足以下核心业务规则：

- 一个用户名只能对应一个用户账号。
- 一个身份证号只能注册一个账号。
- 用户必须登录后才能购票、退票、改签和查看个人订单。
- 购票时余票不足则不允许下单。
- 退票后必须回补余票。
- 改签时必须同时处理原车次和目标车次的余票变化。
- 用户只能操作自己的订单。
- 管理员功能必须与普通用户功能隔离。

## 5. 数据需求分析

### 5.1 用户数据

系统需要存储以下用户信息：

- 用户 ID
- 用户名
- 密码
- 真实姓名
- 身份证号
- 手机号
- 用户角色
- 创建时间

### 5.2 车次数据

系统需要存储以下车次信息：

- 车次 ID
- 车次编号
- 出发站
- 到达站
- 发车时间
- 到达时间
- 票价
- 总票数
- 剩余票数
- 车次状态

### 5.3 订单数据

系统需要存储以下订单信息：

- 订单 ID
- 订单编号
- 用户 ID
- 车次 ID
- 乘车人姓名
- 乘车人身份证号
- 票价
- 订单状态
- 创建时间
- 支付时间

### 5.4 订单状态定义

建议统一采用以下订单状态：

- `UNPAID`：未支付
- `PAID`：已支付
- `REFUNDED`：已退票
- `CHANGED`：已改签
- `CANCELLED`：已取消

## 6. 非功能需求分析

### 6.1 易用性

- 页面操作流程应尽量简洁。
- 用户应能在较少步骤内完成查票和购票。
- 错误提示信息应清晰明确，便于用户理解和修改输入。

### 6.2 性能要求

- 常规查询操作应在可接受时间内返回结果。
- 在小规模课程项目场景下，应支持多用户基本并发访问。
- 购票、退票、改签操作应保证数据一致性，避免出现余票错误。

### 6.3 安全性

- 用户密码不能明文展示。
- 系统应进行基本的身份认证和权限控制。
- 普通用户不能访问管理员接口。
- 对关键输入字段应进行格式校验，防止非法输入。

### 6.4 可维护性

- 前后端接口应统一规范。
- 数据字段命名应清晰一致。
- 主要业务逻辑应模块化，便于后续扩展。

### 6.5 可靠性

- 对无效输入和异常操作应有明确处理机制。
- 系统在操作失败时不应破坏原有数据状态。

## 8. 验收标准

系统完成后，至少应满足以下验收要求：

- 用户可以成功注册并登录。
- 用户可以根据出发地、目的地和日期查询车次。
- 用户可以对有余票的车次成功购票。
- 用户可以查看自己的订单记录。
- 用户可以对符合条件的订单执行退票。
- 用户可以对符合条件的订单执行改签。
- 管理员可以新增、修改、删除车次。
- 系统能够正确维护余票数量和订单状态。

## 9. 风险与约束

### 9.1 项目风险

- 团队开发时间有限，可能影响功能完成度。
- 前后端协作不清晰时，容易出现接口不一致问题。
- 若业务规则设计不完整，可能导致订单状态和余票逻辑错误。

### 9.2 项目约束

- 本项目为课程实践项目，应优先保证核心流程跑通。
- 第一版不接入真实第三方认证和支付服务，可采用模拟方式完成。
- 系统设计需控制复杂度，避免超出团队开发能力范围。

## 10. 结论

Mini-12306 的核心需求较明确，适合按“注册登录、查票、购票、订单管理、管理员维护车次”这一主线逐步实现。第一版应优先保证业务闭环完整，即用户能够完成从注册登录到查询、购票、退票和改签的完整流程，同时管理员能够维护基础车次数据。在此基础上，后续可逐步扩展支付、短信通知、身份认证和移动端等功能。
