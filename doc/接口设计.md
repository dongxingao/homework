# Mini-12306 接口设计

## 1. 文档说明

本文档用于定义 Mini-12306 系统前后端交互接口，作为前端页面开发、后端接口实现和联调测试的统一依据。

本项目第一版采用 `RESTful` 风格接口，数据交换格式统一为 `JSON`。

## 2. 设计目标

接口设计的目标如下：

- 保证前后端对业务功能理解一致
- 保证请求参数和返回结果统一规范
- 支撑注册、登录、查票、购票、退票、改签和管理员车次管理等核心功能
- 降低联调阶段的沟通成本

## 3. 基本约定

### 3.1 接口前缀

系统接口统一使用以下前缀：

```text
/api
```

### 3.2 数据格式

- 请求体格式：`application/json`
- 返回体格式：`application/json`

### 3.3 统一返回结构

所有接口统一返回以下结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

字段说明如下：

- `code`：业务状态码，`0` 表示成功，非 `0` 表示失败
- `message`：提示信息
- `data`：返回数据，失败时可为 `null`

### 3.4 常用状态码约定

| code | 含义 |
|---|---|
| 0 | 请求成功 |
| 400 | 参数错误或业务校验失败 |
| 401 | 未登录或登录失效 |
| 403 | 无权限访问 |
| 404 | 资源不存在 |
| 500 | 服务器内部错误 |

### 3.5 身份认证约定

第一版项目可采用以下任一方式：

- 基于 Session 登录态
- 基于 Token 登录态

为保证文档通用性，本文档默认使用 `Token` 方式说明，即登录成功后返回 `token`，后续请求在请求头中携带：

```text
Authorization: Bearer <token>
```

如果项目实际采用 Session，也可以保持接口路径和参数不变，仅调整认证实现。

## 9. 错误场景说明

系统常见错误场景如下：

### 9.1 用户注册相关错误

- 用户名已存在
- 身份证号已注册
- 手机号已存在
- 参数为空或格式错误

### 9.2 登录相关错误

- 用户名不存在
- 密码错误
- 账号被禁用

### 9.3 购票相关错误

- 用户未登录
- 车次不存在
- 车次已停运
- 车次停止售票
- 余票不足
- 乘车人信息不合法

### 9.4 退票相关错误

- 订单不存在
- 订单不属于当前用户
- 当前订单不可退票

### 9.5 改签相关错误

- 原订单不存在
- 原订单不属于当前用户
- 当前订单不可改签
- 目标车次不存在
- 目标车次余票不足

### 9.6 管理员接口相关错误

- 当前用户不是管理员
- 车次编号重复
- 票数数据不合法
- 目标车次不存在
- 车次已被订单引用，不能删除

## 10. 前后端联调约定

为减少联调问题，建议遵循以下约定：

- 前后端统一字段命名，接口中使用驼峰命名，如 `realName`、`fromStation`
- 时间字段统一格式为 `YYYY-MM-DD HH:mm:ss`
- 金额字段统一保留两位小数
- 所有受保护接口都必须校验登录状态
- 管理员接口必须校验用户角色
- 前后端对订单改签的实现方式必须提前统一

其中，改签逻辑需要明确以下两种方案中的一种：

- 方案一：改签后生成新订单，原订单状态改为 `CHANGED`
- 方案二：直接更新原订单对应的车次信息

从业务清晰度和可追踪性考虑，建议优先采用 `方案一`。
